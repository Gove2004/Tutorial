世界生成

当你开始改写世界生成时，模组就真正拥有了自己的地貌、资源和探索路线。

世界生成的意义 前面的内容大多是在“已有世界”里加东西；而世界生成会直接影响玩家第一次进入世界时看到什么、挖到什么、探索到哪里。这是模组世界观落地的关键层。

为什么世界生成要格外谨慎

因为它发生在世界创建阶段，一旦设计得太激进，玩家整个世界体验都会受影响。你加错一个矿石刷新量，可能让前期 progression 失衡；你塞进太多地形结构，可能把原版地貌破坏得一塌糊涂。

世界生成不是“随便刷点东西”

你要把世界生成理解成一条流程管线。Terraria 在生成世界时会按顺序执行一系列生成任务（GenPass），你的模组需要选择合适时机插入自己的任务，而不是胡乱覆盖原版行为。

ModSystem：世界级逻辑的新入口

较新的 tModLoader 里，许多世界层面的逻辑更适合统一放到 ModSystem 中管理。世界生成、世界状态存档、网络同步、后处理都可以在这里集中组织，这样比把相关逻辑分散在旧教程风格的多个入口里更清晰，也更方便维护。

using Terraria.ModLoader;

namespace ExampleMod.Common.Systems;

public class WorldGenSystem : ModSystem
{
}

虽然这看起来只是一个空壳，但它是后面所有世界生成入口的承载点。

ModifyWorldGenTasks：插入生成步骤

世界生成最常用的入口是 ModifyWorldGenTasks。你可以在这里拿到原版任务列表，然后把自己的任务插到合适的位置。

using Terraria;
using Terraria.IO;
using Terraria.WorldBuilding;
using Terraria.ModLoader;
using System.Collections.Generic;

public class WorldGenSystem : ModSystem
{
    public override void ModifyWorldGenTasks(List<GenPass> tasks, ref double totalWeight)
    {
        int shiniesIndex = tasks.FindIndex(pass => pass.Name.Equals("Shinies"));

        if (shiniesIndex != -1)
        {
            tasks.Insert(shiniesIndex + 1, new PassLegacy("Example Mod Ores", delegate(GenerationProgress progress, GameConfiguration configuration)
            {
                progress.Message = "Placing Star Ore";
                GenerateStarOre();
            }));
        }
    }
}

这里选择插在 Shinies 后面，是因为矿石生成通常和原版矿石生成阶段比较接近。这个位置不是唯一答案，但它反映了一个重要思路：你要尊重原版生成顺序。

注意 别一上来把任务插在最前面或最后面。生成顺序会影响地形覆盖关系。比如你太早放结构，后面可能被原版洞穴冲掉；你太晚放矿石，又可能塞进本不该出现的位置。

用 TileRunner 生成矿石

最经典、也最实用的世界生成练手项目，就是自定义矿石。因为它既能训练你理解生成流程，又容易在游戏里验证结果。

using Terraria;
using Terraria.WorldBuilding;
using Terraria.ModLoader;

private void GenerateStarOre()
{
    for (int i = 0; i < (int)(Main.maxTilesX * Main.maxTilesY * 6E-05); i++)
    {
        int x = WorldGen.genRand.Next(0, Main.maxTilesX);
        int y = WorldGen.genRand.Next((int)WorldGen.rockLayer, Main.maxTilesY - 200);

        WorldGen.TileRunner(
            x,
            y,
            WorldGen.genRand.Next(4, 7),
            WorldGen.genRand.Next(3, 6),
            ModContent.TileType<Content.Tiles.StarOreTile>()
        );
    }
}

TileRunner 的核心作用是：在某个起点附近，以一定规模随机铺出一团矿脉或团块。它特别适合做矿石、结晶、特殊石块等“成片分布”的内容。

矿石生成的几个关键参数

参数作用设计含义
x / y生成起点控制出现在世界哪个深度区域
strength矿脉规模决定团块有多大
steps扩张步数决定矿脉形状复杂度
tileType生成的方块类型即你的自定义矿石 Tile

深度往往比刷新次数更重要。因为矿石的“阶段感”主要来自玩家在哪一层第一次遇到它，而不是它总量有多少。

结构生成：不只是矿石

如果你想做遗迹、祭坛、地下房间、小型神殿，就需要从“随机团块”转向“有结构的放置”。这时更常见的手段是遍历一个区域，按规则放置 Tile 和 Wall。

private void PlaceSimpleShrine(int originX, int originY)
{
    for (int x = originX; x < originX + 12; x++)
    {
        for (int y = originY; y < originY + 8; y++)
        {
            bool isBorder = x == originX || x == originX + 11 || y == originY || y == originY + 7;

            if (isBorder)
            {
                WorldGen.PlaceTile(x, y, ModContent.TileType<Content.Tiles.StarBrickTile>(), mute: true, forced: true);
            }
            else
            {
                WorldGen.KillTile(x, y, false, false, true);
                WorldGen.PlaceWall(x, y, WallID.BlueDungeonUnsafe);
            }
        }
    }
}

这段代码就是最基础的“矩形小神殿”原型。它没有高级 shape 系统，但已经能表达一个封闭房间结构：边框是砖，内部挖空并铺墙。

结构生成的建议 入门先做小型、规则结构，比如 10x6 房间、3x3 水晶簇、固定形状矿脉。别一开始就做超大型迷宫或复杂自适应神殿，不然排查起来非常痛苦。

背景墙与装饰

很多新手世界生成只会放 Tile，结果结构看起来很空。实际上，墙、平台、家具、光源、液体都可以一起生成，才会让区域更像一个真正地点。

WorldGen.PlaceWall(x, y, WallID.BlueDungeonUnsafe);
WorldGen.PlaceTile(x, y, TileID.Torches, mute: true, forced: true);

你可以把这看成“二次装修层”。Tile 决定地形，Wall 和装饰决定氛围。

生成时机怎么选

生成内容不同，插入的时机也不同：

  • 矿石：通常靠近 Shinies。
  • 地表建筑：可放在地表生成之后。
  • 地下结构：更适合洞穴完成后，再找位置插入。
  • 装饰补丁：可放在主结构之后，避免被覆盖。
点击展开：为什么“位置对了但时机错了”也会失败

因为世界生成是多阶段覆盖过程。你可能在正确深度生成了一座神殿，但后面的原版洞穴生成又把它打穿了；或者你先生成矿石，后面的区域替换把矿石刷没了。所以生成逻辑必须和时机一起考虑。

PostWorldGen：后处理阶段

有些事情不适合在主流程里做，而适合在世界生成完成后补处理，比如在宝箱里塞模组物品、调整少量特殊点位、收尾检查。

public override void PostWorldGen()
{
    // 可以在这里修改宝箱内容、补放特殊物品等
}

如果你想给地牢宝箱、地下小屋宝箱加入模组物品，通常就会在这里做，而不是在主地形生成中硬插。

做一个完整的世界生成闭环

1

先做资源矿石

最容易验证，也最能服务 progression。

2

再做小型结构

练习 Tile、Wall、空间清理和放置顺序。

3

最后做世界后处理

往宝箱、特殊点位、奖励房间补内容。

常见坑

  • 矿石刷新量过高，导致 progression 崩坏。
  • 结构生成位置不筛选，刷在地表、空岛或岩浆里。
  • 插入时机不对，被原版后续生成覆盖。
  • 世界生成代码没有考虑大/中/小世界差异。
  • 调试时只看一个世界种子，误以为逻辑稳定。
调试建议 世界生成必须多开新世界验证。至少测试不同大小、不同邪恶类型、不同种子。一个逻辑在某张世界图里没坏，不代表它普遍可用。

本章小结

你现在已经理解了世界生成的核心：通过 ModSystem 接入世界流程，在 ModifyWorldGenTasks 中选对时机插入任务，用 TileRunner 放资源矿石，用遍历与放置 API 做结构，用 PostWorldGen 做后处理。
