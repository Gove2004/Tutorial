物块与家具

从背包物品走向世界本身：让你的模组内容真正成为地图的一部分。

本章定位 物块和家具是 tModLoader 开发中的一个分水岭。做物品时你主要处理背包和使用动作；做物块时，你开始直接改写世界网格、光照、掉落和交互。

Tile 和 Item 是两层东西

很多初学者第一次写可放置物块时会混淆：背包里的“方块物品”和世界里的“方块格子”不是同一个对象。背包里的是 ModItem，负责被玩家拿着、堆叠、消耗；放到世界里以后，真正存在于地图网格中的则是 ModTile。

ModTile

ModTile 用来定义世界中的自定义物块或家具。它关心的是碰撞、光照、挖掘、掉落、帧图、家具交互等地图层面的行为。

因此，可放置物块通常至少有两个类：一个 ModItem 作为背包入口，一个 ModTile 作为世界表现。它们通过 Item.createTile 连接。

创建一个基础矿石物块

矿石是最适合入门的物块类型，因为它有明确的行为：占据格子、阻挡玩家、会被镐子挖掉、掉落对应物品。

using Microsoft.Xna.Framework;
using Terraria;
using Terraria.ID;
using Terraria.ModLoader;

namespace ExampleMod.Content.Tiles;

public class StarOreTile : ModTile
{
    public override void SetStaticDefaults()
    {
        Main.tileSolid[Type] = true;
        Main.tileBlockLight[Type] = true;
        Main.tileSpelunker[Type] = true;
        Main.tileOreFinderPriority[Type] = 410;

        MineResist = 2f;
        MinPick = 55;

        AddMapEntry(new Color(120, 170, 255), CreateMapEntryName());
        HitSound = SoundID.Tink;
        DustType = DustID.BlueCrystalShard;
    }
}

几个关键字段

字段/属性作用说明
Main.tileSolid[Type]是否为实体方块玩家和 NPC 是否会被阻挡
Main.tileBlockLight[Type]是否阻挡光线矿石、石块通常为 true
MineResist挖掘抗性越高越难挖
MinPick最低镐力需求控制玩家什么时候能挖
AddMapEntry地图显示决定小地图上的颜色与名称
注意 SetStaticDefaults() 在新版本中是配置 Tile 静态属性的主要位置。很多旧教程会出现 SetDefaults()，阅读时要留意版本差异。

让物品能放置这个物块

现在我们有了世界中的 Tile，还需要一个能放进背包、能被玩家拿着使用的 Item。

using Terraria;
using Terraria.ID;
using Terraria.ModLoader;

namespace ExampleMod.Content.Items.Placeables;

public class StarOreItem : ModItem
{
    public override void SetDefaults()
    {
        Item.width = 20;
        Item.height = 20;
        Item.maxStack = 9999;
        Item.useTurn = true;
        Item.autoReuse = true;
        Item.useAnimation = 15;
        Item.useTime = 10;
        Item.useStyle = ItemUseStyleID.Swing;
        Item.consumable = true;
        Item.createTile = ModContent.TileType<Content.Tiles.StarOreTile>();
        Item.value = Item.buyPrice(silver: 5);
        Item.rare = ItemRarityID.Blue;
    }
}

这里最关键的是 Item.createTile。这个字段不是说“物品变成了 Tile”，而是说：“玩家使用该物品时，尝试在世界中放置这个 Tile 类型。”

物块掉落：挖掉后掉什么

对于普通方块，默认情况下经常会掉落对应物品。但当你做特殊逻辑时，最好明确理解掉落是怎么控制的。基础做法是在 ModTile 中指定掉落物。

public override IEnumerable<Item> GetItemDrops(int i, int j)
{
    yield return new Item(ModContent.ItemType<Items.Placeables.StarOreItem>());
}

如果只是最基础的方块掉落，你不一定每次都要复杂重写。但一旦你需要“有概率掉多个物品”“受工具影响掉不同东西”“特殊事件期间掉额外材料”，就会用到更精细的掉落控制。

点击展开：什么时候需要自定义掉落
  • 矿石挖掉后掉落矿砂，而不是自身方块。
  • 家具破坏时额外返还装饰材料。
  • 事件期间同一物块掉落特殊货币。
  • 玩家使用特殊工具时改变掉落结果。

家具和普通物块有什么不同

普通物块一般是 1x1 网格，而家具往往占用多个格子，并且有放置方向、底部支撑、交互行为、帧图结构。比如工作台、箱子、床、椅子、门，它们都不仅是“更大的方块”。

家具 Tile

家具通常仍然继承 ModTile，但会通过 TileObjectData 配置尺寸、锚点、方向、放置条件与帧图方式。

一个基础工作台

using Microsoft.Xna.Framework;
using Terraria;
using Terraria.ID;
using Terraria.ModLoader;
using Terraria.ObjectData;

namespace ExampleMod.Content.Tiles.Furniture;

public class StarWorkbenchTile : ModTile
{
    public override void SetStaticDefaults()
    {
        Main.tileTable[Type] = true;
        Main.tileSolidTop[Type] = true;
        Main.tileNoAttach[Type] = true;
        Main.tileFrameImportant[Type] = true;

        TileObjectData.newTile.CopyFrom(TileObjectData.Style2x1);
        TileObjectData.newTile.CoordinateHeights = new[] { 18 };
        TileObjectData.addTile(Type);

        AddToArray(ref TileID.Sets.RoomNeeds.CountsAsTable);
        AdjTiles = new int[] { TileID.WorkBenches };

        AddMapEntry(new Color(80, 120, 200), CreateMapEntryName());
    }
}

AdjTiles 是工作台最关键的配置之一。它告诉配方系统：靠近这个自定义工作台时，也可以视为靠近原版工作台。这样你就能让它参与合成站逻辑。

设计建议 新手做家具，建议先模仿原版同类家具的尺寸和行为。比如工作台先按 2x1，箱子先按原版箱子逻辑，床先按原版床逻辑。先跑通，再做风格化。

与家具配套的放置物品

和矿石方块一样，家具通常也需要一个对应的 ModItem 作为放置入口。否则玩家能在代码里定义出工作台，却没法在游戏里拿到并摆出来。

public class StarWorkbenchItem : ModItem
{
    public override void SetDefaults()
    {
        Item.width = 28;
        Item.height = 18;
        Item.maxStack = 9999;
        Item.useTurn = true;
        Item.autoReuse = true;
        Item.useAnimation = 15;
        Item.useTime = 10;
        Item.useStyle = ItemUseStyleID.Swing;
        Item.consumable = true;
        Item.createTile = ModContent.TileType<Content.Tiles.Furniture.StarWorkbenchTile>();
    }
}

这样一来，前面章节里提到的 StarWorkbenchTile 才真正能通过物品进入玩家手中，并在后续配方或商店示例里被自然引用。

帧图与动画

物块贴图和物品图标不同。物品通常是一张独立图标，而 Tile 贴图往往是按网格帧切出来的。家具尤其依赖帧图，因为不同格子、不同方向、不同状态可能对应不同帧。

对于发光或动画物块，你还可能会接触：

  • Main.tileFrameImportant[Type]：帧是否重要，家具通常为 true。
  • AnimateTile：控制动画帧。
  • AnimationFrameHeight：单帧高度。
  • ModifyLight：控制物块发光。

发光物块示例

public override void ModifyLight(int i, int j, ref float r, ref float g, ref float b)
{
    r = 0.2f;
    g = 0.45f;
    b = 0.9f;
}

发光不要乱给太强。Terraria 的画面层次很依赖局部光照，如果你的矿石像路灯一样亮，整个地下体验会被破坏。

实战路线：自定义矿石 + 熔炉家具

1

先做矿石 Tile

配置实体、阻光、镐力需求、地图颜色。

2

再做矿石 Item

用 createTile 连接到矿石 Tile。

3

做熔炉家具 Tile

用 TileObjectData 配置尺寸，并用 AdjTiles 接入合成站。

4

最后加入配方

让矿石能合成锭，锭能合成家具，形成完整资源闭环。

常见坑

  • 只写了 ModTile，忘了写对应放置物品。
  • createTile 指向错误类型，导致物品无法放置。
  • 家具尺寸和贴图帧不匹配，放置后显示错乱。
  • 忘记配置 AdjTiles，自定义工作台看起来像工作台，却不能当合成站。
  • 过度发光，导致地下环境视觉失衡。
排查建议 物块问题要分层查：先确认 Item 能使用，再确认 createTile 指向正确，再确认 Tile 静态属性，再确认贴图帧。不要一开始就怀疑世界生成或复杂逻辑。

本章小结

物块与家具把你的模组从“背包内容”推进到“世界内容”。你现在应该清楚：背包物品和世界 Tile 是两层对象，放置逻辑靠 createTile 连接，家具则需要更多关于尺寸、帧图和交互的配置。

下一章我们会把前面做过的物品、材料、物块串起来，进入合成配方系统。配方是内容之间形成生态的关键：没有配方，很多内容只是孤立地存在；有了配方，玩家才会沿着你设计的路线推进。
