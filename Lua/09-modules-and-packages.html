09 - 模块系统与包管理

Lua 的模块系统极其简洁：一个文件 = 一个模块 = 一个返回 table 的函数。没有 package.json，没有 import/export 关键字。理解 require 的路径搜索和缓存机制，你就能驾驭任意规模的 Lua 项目。

require 的核心行为

require(modname)

在 package.path 中搜索模块文件，加载并执行，缓存结果到 package.loaded[modname]。同一进程内多次 require 同一模块不会重复执行。

-- 第一次 require：搜索 → 加载 → 执行 → 缓存
local mymod = require("mymod")

-- 第二次 require：直接从 package.loaded 返回缓存
local mymod2 = require("mymod")  -- mymod2 == mymod，同一引用

-- require 的伪代码逻辑
function my_require(name)
    if package.loaded[name] then
        return package.loaded[name]  -- 缓存命中
    end
    local loader, err = search_and_load(name)
    if not loader then
        error(err)
    end
    local result = loader(name)
    if result ~= nil then
        package.loaded[name] = result
    else
        package.loaded[name] = true  -- 模块没返回值，标记已加载
    end
    return package.loaded[name]
end

模块文件的标准写法

推荐写法（现代，Lua 5.2+）旧式写法（module 函数，已废弃）
 
-- mymod.lua
local M = {}

local function private_helper()
    -- 内部函数，不暴露
end

function M.public_func(x)
    return x * 2
end

M.VERSION = "1.0.0"

return M  -- 返回 table
 
-- 旧式（Lua 5.1 的 module 函数）
module("mymod", package.seeall)
function public_func(x)
    return x * 2
end
-- 不推荐：污染全局命名空间
为什么 module() 被废弃？ module("foo", package.seeall) 把模块的全局环境改成了 _G——模块内部可以访问任何全局变量，失去了封装性。现代写法用 local table + return 保证隔离。

package.path：require 的查找路径

-- 查看当前的搜索路径
print(package.path)
-- 典型输出（Linux）:
-- ./?.lua;/usr/local/share/lua/5.4/?.lua;/usr/local/share/lua/5.4/?/init.lua

-- ? 代表模块名。require("mymod") 会依次尝试:
-- ./mymod.lua
-- /usr/local/share/lua/5.4/mymod.lua
-- /usr/local/share/lua/5.4/mymod/init.lua

-- 动态添加搜索路径
package.path = package.path .. ";./libs/?.lua"

-- 或者只添加特定路径
package.path = "./src/?.lua;" .. package.path  -- 优先搜索 src 目录

-- 对于 C 模块（.so/.dll），对应的是 package.cpath
print(package.cpath)
-- 典型: ./?.so;/usr/local/lib/lua/5.4/?.so

package.loaded：模块缓存

-- 查看已加载的模块
for name, module in pairs(package.loaded) do
    print(name, type(module))
end

-- 强制重新加载模块（绕过缓存）
package.loaded["mymod"] = nil
require("mymod")  -- 重新加载

-- 热重载（hot reload）的基本原理
local function hot_reload(modname)
    package.loaded[modname] = nil  -- 清除缓存
    return require(modname)
end

_ENV 与沙盒

Lua 5.2+ 引入 _ENV 变量——每个函数可以有自己独立的全局环境。

-- _ENV 是每个 chunk（文件/函数）的局部变量，默认值是 _G
-- 修改 _ENV 可以改变这个 chunk 看到的「全局变量」

-- 创建沙盒环境
local sandbox_env = {
    print = print,
    math = math,
    string = string,
    table = table,
    pairs = pairs,
    ipairs = ipairs,
    type = type,
    error = error,
    tostring = tostring,
}

-- 在沙盒中执行代码
local function sandbox_execute(code)
    local fn, err = load(code, "sandbox", "t", sandbox_env)
    if not fn then return nil, err end
    return pcall(fn)
end

sandbox_execute([[
    print(math.sqrt(16))          -- 4，math 在白名单中
    -- os.exit()                  -- 报错：os 不在沙盒环境中
    -- file = io.open("secret")   -- 同样不可访问
]])

-- 安全沙盒必须做的事情：
-- 1. 移除 debug 库（防止反沙盒）
-- 2. 移除 loadfile/dofile/require（防止加载外部代码）
-- 3. 限制 string.dump（防止 dump 字节码）
-- 4. 考虑 string.rep 的 DoS 攻击（无限重复字符串）

工程化：多模块项目目录结构

-- 推荐的项目布局
-- project/
-- ├── main.lua           -- 入口
-- ├── src/
-- │   ├── init.lua       -- 项目命名空间
-- │   ├── core/
-- │   │   ├── engine.lua
-- │   │   └── config.lua
-- │   └── utils/
-- │       ├── math.lua
-- │       └── string.lua
-- └── libs/              -- 第三方（手动管理或 LuaRocks）
--     └── ...

-- src/init.lua
local project = {}
project.core = require("src.core.engine")
project.utils = require("src.utils.math")
return project

-- 设置路径（在 main.lua 开头）
package.path = "./src/?.lua;./src/?/init.lua;" .. package.path

-- 现在可以这样用
local project = require("src.init")
project.core.start()

3

安全研究

分析上面的沙盒实现有什么安全漏洞。提示：思考 string.rep 的 DoS、getmetatable 访问字符串元表、以及 debug 库通过其他手段获取（例如从函数的 upvalue 中拿到）。

本章小结

  • require 搜索 → 加载 → 缓存到 package.loaded。同进程同模块只执行一次
  • 现代模块写法：local M = {}; ...; return M。废弃 module() 函数
  • package.path（Lua 模块）和 package.cpath（C 模块）控制搜索路径
  • _ENV 实现沙盒——限制代码可见的「全局变量」，是安全执行用户代码的基础
  • LuaRocks 是 Lua 的包管理器，类似于 pip/npm/cargo
