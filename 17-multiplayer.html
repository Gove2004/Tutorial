多人联机同步

单机里能跑，不等于联机里就对。多人同步是模组从“自己能玩”走向“别人也能一起玩”的关键门槛。

这一章非常重要 很多模组在单机测试时看起来完全正常，但一到联机就出现：弹幕重复、Boss 状态错乱、UI 数据不同步、玩家专属资源只自己看得到。原因通常不是系统本身写错，而是同步思路没建立起来。

先建立一个观念：不是所有数据都会自动同步

Terraria 和 tModLoader 确实会替你同步很多基础内容，比如玩家位置、原版生命值、基础弹幕实体等。但你自己加的世界标志、自定义资源值、特殊技能状态、Boss 阶段附加数据，不会凭空自动同步给所有人。你必须明确决定：什么数据需要同步，谁来发，什么时候发。

同步策略

多人联机同步不是“把所有变量都发出去”，而是先判断哪些状态必须跨客户端一致，再决定是由服务端广播，还是由拥有者上报，还是在特定事件点手动同步。

最常见的同步工具：ModPacket

tModLoader 中自定义数据同步最常见的入口就是 ModPacket。你创建一个数据包，把必要的信息写进去，再在接收端解析并应用。

using System.IO;
using Terraria;
using Terraria.ID;
using Terraria.ModLoader;

namespace ExampleMod;

public class ExampleMod : Mod
{
    internal enum MessageType : byte
    {
        SyncStarEnergy
    }

    public override void HandlePacket(BinaryReader reader, int whoAmI)
    {
        MessageType messageType = (MessageType)reader.ReadByte();

        switch (messageType)
        {
            case MessageType.SyncStarEnergy:
                byte playerIndex = reader.ReadByte();
                int energy = reader.ReadInt32();
                Main.player[playerIndex].GetModPlayer<StarPlayer>().starEnergy = energy;

                if (Main.netMode == NetmodeID.Server)
                {
                    ModPacket packet = GetPacket();
                    packet.Write((byte)MessageType.SyncStarEnergy);
                    packet.Write(playerIndex);
                    packet.Write(energy);
                    packet.Send(-1, whoAmI);
                }
                break;
        }
    }
}

你可以看到，包的结构非常明确：先写消息类型，再写玩家索引，再写具体数值。接收时按同样顺序读回来。这种“写入顺序必须和读取顺序完全一致”的原则，是联机同步的基本功。上面的示例还补上了一个很关键的细节：如果服务器收到客户端上报的数据，往往还需要再广播给其他客户端，否则只有服务端自己知道状态更新了，别的玩家却看不到。

注意 包里字段顺序不一致，是多人同步最经典也最隐蔽的 bug 来源之一。写的时候是什么顺序，读的时候就必须完全一样。

什么时候发包

不是每一帧都需要同步，也不应该每一帧都同步。你要挑状态真正改变的时候发，比如资源值变化、开关状态切换、Boss 进入下一阶段、某个技能正式触发。

public void SyncStarEnergy(Player player)
{
    ModPacket packet = Mod.GetPacket();
    packet.Write((byte)ExampleMod.MessageType.SyncStarEnergy);
    packet.Write((byte)player.whoAmI);
    packet.Write(starEnergy);
    packet.Send();
}

这类写法适合在资源值发生关键变化后调用，而不是放进每帧更新里。否则多人环境下很容易因为发包过频导致带宽浪费或状态抖动。

谁来发：拥有者还是服务端

多人同步最容易混乱的点之一，就是“到底该谁发”。一个简单经验是：

  • 玩家自己的输入或本地状态变化，通常由该玩家上报。
  • 真正决定世界结果的状态，通常由服务端确认并广播。
  • 世界级数据和 Boss 关键阶段，优先以服务端为准。

如果你让每个客户端都自己决定 Boss 是否切二阶段，结果一定乱；如果你让服务端来裁定，然后广播结果，大家才会看到一致状态。

联机思维的核心 永远先问自己：这个状态的“权威来源”是谁？答案不清楚，同步设计通常也不会清楚。

自定义玩家数据同步示例

假设你给玩家做了一套“星辉能量”资源条。单机里只要本地变量能涨能降就够了；联机里，别的玩家和服务端也需要知道这个值。

using Terraria;
using Terraria.ID;
using Terraria.ModLoader;

namespace ExampleMod.Common.Players;

public class StarPlayer : ModPlayer
{
    public int starEnergy;

    public override void ResetEffects()
    {
        if (starEnergy < 0)
            starEnergy = 0;
    }

    public void GainEnergy(int amount)
    {
        starEnergy += amount;

        if (Main.netMode == NetmodeID.SinglePlayer)
            return;

        SyncStarEnergy(Player);
    }

    private void SyncStarEnergy(Player player)
    {
        ModPacket packet = Mod.GetPacket();
        packet.Write((byte)ExampleMod.MessageType.SyncStarEnergy);
        packet.Write((byte)player.whoAmI);
        packet.Write(starEnergy);
        packet.Send();
    }
}

这套结构就已经足够说明同步思路：资源变化时发包，而不是每帧都刷；数据包里只带必要最小信息；接收端再把值写回对应玩家。

自定义世界数据同步

如果同步的是世界级数据，比如某个 Boss 是否已击败、某扇门是否开启、某区域是否被净化，那就更适合挂在 ModSystem 上，并配合世界存档与网络同步一起处理。

public override void NetSend(BinaryWriter writer)
{
    writer.Write(downedStarWarden);
}

public override void NetReceive(BinaryReader reader)
{
    downedStarWarden = reader.ReadBoolean();
}

这类同步通常在客户端加入世界时尤其重要，因为新加入的人需要立刻知道当前世界已经处于什么进度，而不是等某事件再次触发才知道。

自定义弹幕和 NPC 的联机注意点

前面你已经写过自定义弹幕和敌怪。联机里最常见的问题，是实体重复生成。因为如果客户端和服务端都执行了“生成一发弹幕”这件事，最终场上就会出现两发。

if (Main.netMode != NetmodeID.MultiplayerClient)
{
    Projectile.NewProjectile(...);
}

这类判断你前面已经见过。它的本质是：涉及生成世界实体时，要明确权威端，避免重复创建。

调试联机最实用的方法

最常见也最实用的方式，就是本机双开：一个开主机，一个开客户端，使用 localhost 连接。这样你能快速重现“自己看得见，别人看不见”或“别人已经进入二阶段，我这里还没切”的问题。

1

先在单机验证机制

确保逻辑本身是对的。

2

本机双开做最小联机验证

只测一个功能，例如资源同步或 Boss 状态切换。

3

观察双方视角是否一致

重点看数值、弹幕、阶段、UI 是否完全同步。

4

最后再做复杂场景联测

比如多人同时攻击、多人同时触发技能或切地图。

同步设计最常见的错误

  • 以为自定义变量会自动同步。
  • 所有客户端都自己决定关键状态，导致分歧。
  • 发包过于频繁，性能和稳定性都受影响。
  • 生成实体时没有区分服务端和客户端，导致重复生成。
  • HandlePacket 读写顺序不一致。
一个非常实用的判断标准 问自己：如果另一个玩家只看他自己的屏幕，他是否能看到和我这里完全一致的结果？如果不能，那你的机制在多人环境里就还没算真正完成。

本章小结

多人同步的本质不是记住几个 API，而是建立一套权威思维：谁拥有状态、谁负责广播、何时发包、哪些数据真的需要同步。你现在已经掌握了最常见的自定义同步方式，也知道了世界级数据、玩家级数据和实体生成逻辑分别该怎么处理。
