模组结构与生命周期

理解项目骨架和加载顺序，是后面所有内容开发的地基。

这一章的目标 你不只是知道“文件放哪”，而是要理解：为什么放这里、谁会在什么时候被加载、哪些类是内容类，哪些类更像系统类。

先看模组的最小骨架

当你创建一个新模组时，目录通常不会很复杂，但里面已经隐含了几种不同职责的文件。最核心的有三类： 入口类、元数据文件、内容类。入口类定义模组本身；元数据文件帮助构建与发布；内容类则是你真正往游戏里加东西的地方。 

using Terraria.ModLoader;

namespace ExampleMod;

public class ExampleMod : Mod
{
    public override void Load()
    {
        // 模组加载时执行
    }

    public override void Unload()
    {
        // 模组卸载时清理静态引用
    }
}

这个 Mod 子类是整个模组的入口。你可以把它看成“应用程序的主上下文”，尽管很多具体内容并不会都写在这里。 绝大多数时候，它更适合放全局初始化、资源注册、跨系统协调，而不是塞满各种物品逻辑。

Mod 类

每个模组通常有且至少有一个继承自 Mod 的类。它代表这个模组本身，持有模组级别的上下文。 当你需要访问模组资产、注册全局行为、读取模组元信息时，通常都绕不开它。

推荐的目录组织方式

tModLoader 并不强制你必须用某个目录结构，但“能跑”和“可维护”是两回事。一个 3 个文件的小模组随便放都行， 可一旦你做武器、NPC、弹幕、UI、世界生成，目录不分层很快就会乱到没法找东西。 

ExampleMod/
├── ExampleMod.cs
├── build.txt
├── description.txt
├── Common/
│   ├── Players/
│   ├── Systems/
│   └── Utilities/
├── Content/
│   ├── Items/
│   ├── Projectiles/
│   ├── NPCs/
│   ├── Buffs/
│   ├── Tiles/
│   └── Biomes/
├── Effects/
├── UI/
└── Assets/

为什么很多社区项目喜欢用 Common 和 Content 分层

这是一个非常实用的习惯。Content 目录放“真正进入游戏内容表的对象”，比如物品、NPC、弹幕、Buff； Common 则放辅助逻辑，比如自定义玩家数据、通用系统、工具函数、共享常量。这样你一眼就能分辨： 某个类到底是在“注册新内容”，还是在“支撑这些内容运行”。

经验建议 如果一个类本身不会作为游戏内容直接出现，而只是为其他内容服务，那它更适合放进 Common 或 Systems，而不是硬塞进 Content。

Autoload 机制：为什么你写了类，它就自己进游戏了

很多初学者第一次看到 ModItem 时都会疑惑：我明明没写“注册物品”的代码，为什么它已经生效了？答案就在 Autoload 机制里。 tModLoader 会扫描程序集中的特定类型，只要它们符合加载条件，就会自动注册到内容系统中。

Autoload

Autoload 就是自动加载机制。它会帮你把符合规则的类自动注册为内容对象，因此你通常不用手写一份“把每个物品一个个 Add 进去”的清单。

哪些类型通常会被自动加载

  • ModItem：物品
  • ModProjectile：弹幕
  • ModNPC：敌怪 / Boss / 城镇 NPC
  • ModBuff：Buff 与 Debuff
  • ModTile：物块与家具
  • ModSystem：系统级逻辑

也正因为有 Autoload，你会发现 tModLoader 项目更像“约定优于配置”。只要你按框架约定写类、放资源、使用正确命名，很多事情它都会自动接起来。

点击展开：什么时候会想要关闭或绕开自动加载

比如你在做实验性内容、跨版本兼容、条件注册，或者某个类需要按配置决定是否启用。这时你就会更在意哪些类默认会被扫描，哪些内容需要手动控制。 初学阶段不用太急着深挖，但你要知道自动加载不是“魔法”，只是框架层做了扫描和注册。

ModContent：从“类”走到“已注册内容”的桥

写代码时你经常会遇到一个问题：类已经定义了，但游戏内部很多地方需要的是数值 ID、类型索引或已注册的内容引用。 ModContent 就是用来解决这个问题的静态入口。 

int itemType = ModContent.ItemType<Content.Items.DebugSword>();
int projectileType = ModContent.ProjectileType<Content.Projectiles.StarBolt>();
int buffType = ModContent.BuffType<Content.Buffs.RageBuff>();

你可以把它理解为“向 tModLoader 索取已注册内容的官方方式”。它不是创建对象，而是告诉框架： “请把这个类对应的运行时类型 ID 给我，我要把它填到别的配置里。”

注意 这里返回的通常不是 C# 意义上的实例对象，而是内容类型索引。很多 API 需要的就是这个索引，而不是你自己 new 出来的对象。

加载阶段：理解生命周期比背方法名更重要

tModLoader 的一大特点，是它不是“所有东西一次性跑完”，而是分阶段加载。不同阶段适合做不同的事。 如果你在错误时机做初始化，最轻是拿到 null，最重是整个模组加载失败。

你可以先记住这几个关键阶段

阶段作用适合做什么
Load模组资源和类型开始装载注册系统、读取资产、初始化静态内容
PostSetupContent所有内容都已基本注册完成引用其他模组内容、做跨模组检查
AddRecipes / PostAddRecipes配方系统建立阶段添加或调整配方
Unload模组卸载清理静态引用，防止热重载残留

社区里常会把加载顺序讲得很细，但你前期真正需要掌握的是：不要过早依赖尚未注册完成的内容，不要忘记清理静态引用。 这两个原则能帮你避开很多诡异错误。 

public class ExampleMod : Mod
{
    public static object CachedReference;

    public override void Load()
    {
        CachedReference = new object();
    }

    public override void Unload()
    {
        CachedReference = null;
    }
}

为什么 Unload 很重要

因为游戏和模组开发都经常涉及热加载、重新加载、切换世界。只要你保留了不该保留的静态引用，就可能导致旧对象挂住、数据脏掉、资源释放不完整。 很多“明明重载了模组却还像旧逻辑在运行”的问题，本质上都和清理不彻底有关。

元数据文件：别把它当成无聊配置

除了 C# 代码，模组通常还会有 build.txt 和 description.txt 这类文件。它们决定了模组显示名、作者、版本、简介、依赖等信息， 同时也会影响构建和发布体验。 

displayName = Example Mod
author = YourName
version = 0.1

刚开始你可能只填最基本的信息，但随着项目变大，这些元数据会越来越重要。比如版本号会影响玩家更新认知，依赖信息会影响加载顺序与兼容性判断。

常见错误 很多人重命名模组时只改了显示名，没改内部目录、命名空间和资源路径，结果导致贴图丢失、引用错乱。内部名一旦稳定，尽量少反复改。

给新手的一套可维护习惯

如果你准备认真做一个模组，而不是只写几个实验文件，我建议从第二章开始就养成下面这套习惯：

  1. 内容类和系统类分目录放。
  2. 一个类尽量只做一种职责。
  3. 类名、贴图文件名、资源路径保持一致。
  4. 需要全局共享的逻辑放 Common，不要复制粘贴到每个内容类。
  5. 凡是用了静态缓存，就考虑 Unload 如何清理。

这些听起来像工程洁癖，但实际上它们会直接影响你三周后还能不能看懂自己写过什么。

本章小结

现在你应该已经知道，一个 tModLoader 模组并不是“随便堆 C# 文件”的集合，而是一个有入口类、有自动加载机制、有生命周期分层、有内容注册系统的完整工程。 这套结构理解透了，后面每加一个物品、每写一个 NPC，你都会更清楚自己是在把内容挂到哪一层。
