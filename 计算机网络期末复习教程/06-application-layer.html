第6章 应用层

应用层是TCP/IP体系中最贴近用户的一层——用户每天都在和使用应用层协议的应用程序打交道：用浏览器看网页（HTTP）、发邮件（SMTP/POP3/IMAP）、访问网站（DNS）、传文件（FTP）、自动获取IP地址（DHCP）。这一章我们来理解这些日常使用的服务"背后"发生了什么。和前面几章一样，每个协议我们都会从为什么需要它开始，再讲它是什么，最后看实际怎么工作。

本章考点

  • HTTP请求报文和响应报文的结构、HTTP状态码分类
  • HTTP/1.0 vs 1.1 vs 2.0 的区别

  • FTP主动与被动模式的区别

6.1 域名系统DNS

为什么需要DNS？

想一想：当你想要访问一个网站时，你在浏览器地址栏里输入的是什么？是 www.example.com，而不是 93.184.216.34。但网络层的IP协议只知道IP地址，不知道域名。所以我们需要一种机制，把人类易记的域名转换成机器能处理的IP地址。

这就是DNS（Domain Name System，域名系统）要解决的问题——它是一个分布在互联网上的层次化的"翻译服务"。说"层次化"是因为：如果让一台服务器记住全世界所有域名和IP的对应关系，这台服务器会被压垮。DNS的解决方案是把这个任务分摊给不同层次的服务器来共同完成。

DNS的核心功能

将域名（如 www.example.com）解析为IP地址（如 93.184.216.34）。这是一个分布式、层次化的数据库系统，全世界没有任何一台服务器存有全部域名信息。

域名的层次结构

域名是从右到左阅读的层次结构。以 www.example.com 为例：

.

根域名（Root）

最顶层，用一个"点"表示，通常省略不写。根域名服务器是DNS体系的"最高权威"。

.com

顶级域名（Top-Level Domain, TLD）

包括通用顶级域名（.com, .org, .edu, .net等）和国家顶级域名（.cn, .jp, .uk等）。

example

二级域名（Second-Level Domain）

组织或个人注册的域名。example.com的所有者拥有这个二级域名的管理权。

www

三级域名（Third-Level Domain）/ 主机名

二级域名所有者可以在自己域名下创建任意多个三级域名，如 www、mail、ftp等。

顶级域名分为三类：

  • 通用顶级域名（gTLD）：.com（商业）、.org（非营利组织）、.net（网络服务）、.edu（教育）、.gov（政府）等
  • 国家顶级域名（ccTLD）：.cn（中国）、.us（美国）、.jp（日本）、.uk（英国）等
  • 基础结构域名：.arpa，用于反向DNS解析（从IP地址反查域名）

域名服务器的层次

和域名的层次结构相对应，域名服务器也分为四个层次：

层次谁管理知道什么特点
根域名服务器ICANN授权的机构所有顶级域名服务器的IP地址全球13组（用A~M编号），每组有多个镜像。是DNS查询的"入口"。
顶级域名服务器各TLD的管理机构该TLD下所有二级域名的权威服务器地址如. com的TLD服务器知道所有.com域名的权威服务器在哪。
权威域名服务器域名所有者或DNS服务商该域名下所有DNS记录（A记录、MX记录、CNAME记录等）每个域名的最终"答案"就在这里。
本地域名服务器ISP或企业网络管理员缓存了最近查询过的域名记录用户主机直接访问的DNS服务器，是解析的"第一站"。

DNS解析过程：递归查询与迭代查询

当你在浏览器输入 www.example.com 并回车时，背后发生了一系列的DNS查询。理解这个流程，首先要明确两种查询方式：

对比维度递归查询迭代查询
工作方式你问一个人，他负责帮你找到答案，中间他问了谁你不关心你问一个人，他说"我不知道，你去问那个人"，你再去问下一家
谁的负担大被查询的服务器（它要替你把活干完）客户端（你要自己跑多家）
客户端复杂度简单——问一次就拿到结果复杂——要多次查询
典型用法主机对本地DNS服务器本地DNS服务器对根/顶级/权威服务器

实际的DNS解析过程通常是两种方式的混合使用。以下是完整的流程：

1

主机查询本地DNS服务器（递归查询）

你的电脑向本地DNS服务器（可能是你的路由器、或者ISP的DNS如 8.8.8.8）发出查询："www.example.com的IP是什么？" 这是递归查询——本地DNS服务器必须给出最终答案。

2

本地DNS查询根服务器（迭代查询）

本地DNS服务器检查自己的缓存，没找到。于是问根服务器："www.example.com的IP是什么？" 根服务器说："我不知道，但我告诉你.com的TLD服务器在哪里。"（返回.com TLD服务器的IP地址）这是迭代查询——根服务器没有替本地DNS去查，而是给了"下一站"的地址。

3

本地DNS查询TLD服务器（迭代查询）

本地DNS服务器问.com的TLD服务器："www.example.com的IP是什么？" TLD服务器说："我不知道，但我知道example.com的权威服务器在哪里。"（返回example.com权威服务器的IP地址）

4

本地DNS查询权威服务器（迭代查询）

本地DNS服务器问example.com的权威服务器："www.example.com的IP是什么？" 权威服务器查了一下自己的DNS记录，回答："www.example.com的IP是93.184.216.34。"

5

返回最终结果

本地DNS服务器把得到的IP地址（93.184.216.34）返回给你的电脑。你的电脑用这个IP地址和www.example.com的服务器建立TCP连接。

这个过程中，主机只发了一次查询（步骤1），剩下的查询工作都由本地DNS服务器代为完成。主机不需要知道根服务器在哪、TLD服务器在哪——它只需要知道本地DNS服务器的地址就够了。

DNS高速缓存

每次DNS查询都需要经过根服务器→TLD服务器→权威服务器这个链条吗？如果真是这样，根服务器和TLD服务器会累垮。实际上，DNS大量使用缓存（Caching）来减少查询次数：

  • 本地DNS服务器会把查询过的域名记录缓存一段时间
  • 每条缓存记录都有一个TTL（生存时间），过期后自动删除
  • 如果再次收到同样域名的查询，直接从缓存返回结果，不用再去根服务器和TLD服务器绕一圈
  • 主机本身也有DNS缓存（操作系统和浏览器各自维护一个DNS缓存）

这就是为什么你访问一个网站后，第二次访问通常比第一次快——因为DNS解析这一步被缓存跳过了。

6.2 文件传输协议FTP

为什么需要FTP？

在网络中传输文件是一个基本需求。HTTP也可以传输文件，但HTTP的设计初衷是获取网页资源——它是一种"拉"（Pull）协议。FTP是专门为文件传输设计的，它既支持"拉"（从服务器下载文件）也支持"推"（上传文件到服务器）。

FTP的特殊之处在于：它使用两个并行的TCP连接来完成工作——一个负责传命令，一个负责传数据。为什么要分开？因为命令和数据在性质上完全不同：命令很短（几条指令）、需要可靠传输；数据可能很大（整个文件）、传输时间长。如果把命令和数据混在同一个连接里，数据传输出错或延迟时，命令也会被卡住。

连接端口用途生命周期
控制连接21传输命令和应答（登录、目录浏览、文件操作命令）整个会话期间持续存在
数据连接20（主动模式）/ 随机端口（被动模式）传输实际的文件数据每次文件传输时建立，完成后关闭

主动模式 vs 被动模式

FTP的数据连接可以在两种模式下建立，它们的核心区别是：谁主动发起数据连接。

对比维度主动模式（Active / PORT）被动模式（Passive / PASV）
数据连接发起方服务器主动连接客户端客户端主动连接服务器
数据连接方向服务器:20 → 客户端:随机端口客户端:随机端口 → 服务器:随机端口
对客户端的要求客户端必须开放随机端口等待服务器连接客户端不需要开放额外端口
穿越防火墙/NAT困难（防火墙通常阻止外来的主动连接）容易（连接由客户端主动发起，防火墙允许）
使用频率较少（主要是历史原因）广泛使用（更安全、更兼容）

具体的交互过程是这样的：

主动模式：客户端通过控制连接告诉服务器"我在某个端口等着你连"（PORT命令）。服务器从自己的20端口向客户端指定的端口发起数据连接。

被动模式：客户端通过控制连接告诉服务器"我想用被动模式"（PASV命令）。服务器回复"好的，我在某个端口等着你连"。客户端向服务器的这个端口发起数据连接。

为什么被动模式更常用？想象一个场景：客户端在公司内网，前面有防火墙和NAT。主动模式下，服务器要主动连客户端的某个端口——防火墙看到"外来的连接请求"，十有八九会拦下来。被动模式下，客户端主动连服务器——防火墙认为这是"对外的请求的回应"，通常会放行。这就是被动模式在现代网络中占主导地位的原因。

用户认证

FTP支持两种访问方式：

  • 账户认证：用户在FTP服务器上有账户，通过用户名和密码登录，拥有对应的读写权限
  • 匿名FTP：使用 "anonymous" 作为用户名，任意邮箱地址作为密码。通常只能下载，不能上传。公共文件下载（如开源软件镜像站）多用此方式
FTP的安全问题：传统FTP的所有数据（包括用户名和密码）都是明文传输的。这意味着在同一个网络中的任何人都可以抓包看到你的密码。安全替代方案：SFTP（基于SSH的文件传输，22端口）或FTPS（FTP over SSL/TLS）。

TFTP：简化版FTP

TFTP（Trivial File Transfer Protocol，简单文件传输协议）是FTP的"轻量版"：

  • 基于UDP而不是TCP（所以不可靠，需要应用层自己处理重传）
  • 没有用户认证机制
  • 只能读写文件，不能列出目录
  • 实现代码量极小，适合嵌入在路由器、交换机等网络设备的固件中
  • 使用69端口

6.3 万维网与HTTP

为什么需要HTTP？

万维网（World Wide Web）的核心需求很简单：用户想通过"点击链接"的方式，在互联网上获取各种资源（网页、图片、视频等）。这个需求背后有几个基本问题要解决：

  • 如何唯一标识一个资源？→ URL（统一资源定位符）
  • 如何描述资源的内容？→ HTML（超文本标记语言）
  • 如何传输资源？→ HTTP（超文本传输协议）

URL格式

URL（Uniform Resource Locator）的通用格式：
<协议>://<主机名>:<端口号>/<路径>
示例：https://www.example.com:443/index.html
其中协议是http或https，主机名是域名或IP地址，端口号可省略（HTTP默认80，HTTPS默认443）。

HTTP报文格式

HTTP的核心是"请求-响应"模式：客户端发送一个请求报文，服务器返回一个响应报文。两种报文的结构非常相似，都由四部分组成。

HTTP请求报文

GET /index.html HTTP/1.1          ← 请求行（方法 + URL + HTTP版本）
Host: www.example.com              ← 首部行
User-Agent: Mozilla/5.0
Accept: text/html, application/json
Accept-Language: zh-CN
Connection: keep-alive
                                   ← 空行（CRLF，分隔首部和实体体）
                                   ← 实体体（GET请求通常为空）

请求行中的方法（Method）告诉服务器客户端想做什么：

  • GET：请求读取URL标识的资源（最常用）
  • HEAD：只请求资源的首部（不返回实体体），用于检查资源是否存在或获取元信息
  • POST：向服务器提交数据（如表单提交、上传文件），数据放在实体体中
  • PUT：上传文件到指定路径（替换已有资源）
  • DELETE：删除指定资源
  • OPTIONS：查询服务器支持哪些方法

HTTP响应报文

HTTP/1.1 200 OK                    ← 状态行（HTTP版本 + 状态码 + 短语）
Date: Thu, 12 Jun 2026 08:00:00 GMT  ← 首部行
Server: Apache/2.4.41 (Unix)
Content-Type: text/html; charset=UTF-8
Content-Length: 1234
Last-Modified: Wed, 11 Jun 2026 12:00:00 GMT
                                   ← 空行
<!DOCTYPE html>                    ← 实体体（实际资源内容）
<html>
<head><title>Example</title></head>
<body><h1>Hello World</h1></body>
</html>

HTTP状态码（高频考点）

状态码是服务器对请求处理结果的数字表示，分为五大类。考试中需要重点记忆的是加粗的部分：

类别范围含义必须记住的状态码
信息性1xx请求已收到，正在处理中100 Continue（客户端应继续发送请求体）
成功2xx请求成功处理200 OK（成功，返回请求的资源）、204 No Content（成功但无内容返回）
重定向3xx需要进一步操作301 Moved Permanently（永久重定向）、302 Found（临时重定向）、304 Not Modified（缓存有效）
客户端错误4xx客户端请求有误400 Bad Request（请求格式错误）、403 Forbidden（禁止访问）、404 Not Found（资源不存在）
服务器错误5xx服务器处理失败500 Internal Server Error（服务器内部错误）、502 Bad Gateway（网关错误）、503 Service Unavailable（服务暂时不可用）
考试必记：200（成功）、301（永久重定向→搜索引擎更新URL）、302（临时重定向）、304（缓存未过期→使用本地缓存）、400（请求格式错了）、403（有权限问题）、404（经典"找不到"）、500（服务器崩了）、503（服务器太忙了）。

HTTP版本演进

对比维度HTTP/1.0（1996）HTTP/1.1（1999）HTTP/2.0（2015）
连接方式非持续连接：每个请求新建一个TCP连接持续连接：默认keep-alive，多个请求复用同一个TCP连接持续连接+多路复用
请求方式串行：发一个、等一个、再发下一个串行+可选流水线：可以连续发请求，但响应必须按序返回并行多路复用：一个连接内同时处理多个请求和响应
数据传输文本格式文本格式二进制帧（更高效）
首部压缩无无HPACK压缩（减少重复首部开销）
服务器推送不支持不支持支持（服务器可主动推送资源）
Host字段可选必须必须（兼容1.1语义）

为什么HTTP/1.1最大的改进是持续连接？我们来算一笔账：一个网页平均需要加载80个资源（HTML、CSS、JS、图片等）。在HTTP/1.0中，每个资源都需要建立和关闭一次TCP连接——80次三次握手+80次四次挥手，光连接管理的开销就大得惊人。HTTP/1.1允许多个请求复用同一个TCP连接，大幅减少了开销。

但HTTP/1.1还有一个问题：队头阻塞（Head-of-Line Blocking）。在一个持续连接上，如果第一个请求的响应还没回来，后面的所有请求虽然已经发出去了，但它们的响应必须排队等待。HTTP/2.0用多路复用解决了这个问题——多个请求和响应可以在同一个连接中交错传输，互不阻塞。

持续连接：非流水线 vs 流水线

  • 非流水线方式：客户端收到一个响应后，才能发出下一个请求。缺点是每获取一个资源就浪费了一个RTT的等待时间。
  • 流水线方式：客户端可以连续发出多个请求，不必等前面的响应回来。服务器按请求到达的顺序依次返回响应。优点是减少了空闲等待时间，但队头阻塞问题仍然存在——如果第一个响应很大，后面的响应即使已经准备好了，也要等前面传完。

Cookie与Session

HTTP是无状态的协议——每个请求都是独立的，服务器不记得上一次请求是谁发的。但很多应用需要"记住"用户：购物车里的商品、登录状态、个性化设置……

Cookie就是在客户端（浏览器）保存状态信息的技术：

  1. 用户登录后，服务器在响应报文中添加 Set-Cookie 首部，内容可能是用户的身份标识或会话ID
  2. 浏览器收到后，把Cookie保存到本地
  3. 以后每次访问同一个网站，浏览器自动在请求报文中携带 Cookie 首部
  4. 服务器根据Cookie中的信息识别用户，"记住"用户的状态
Cookie vs Session：Cookie保存在客户端（浏览器中），Session数据保存在服务器端。通常的工作方式是：服务器生成一个Session ID，通过Cookie发送给客户端，客户端在后续请求中回传这个Session ID，服务器根据Session ID找到对应的Session数据。

代理服务器与Web缓存

代理服务器（Proxy Server）是客户端和服务器之间的"中间人"。它最常见的用途是Web缓存（Web Cache）：

工作原理：

  1. 客户端第一次请求某个资源时，代理服务器代表客户端向目标服务器发起请求
  2. 代理服务器把返回的资源缓存到本地
  3. 另一个客户端请求同一个资源时，代理服务器直接返回缓存的副本，不用再去源服务器获取

Web缓存的好处：

  • 减少延迟：缓存距离客户端更近，响应更快
  • 节省带宽：源服务器不需要重复传输相同的内容
  • 减轻源服务器负载：大量请求在代理层就被处理了

代理服务器的其他用途：过滤和访问控制（企业限制员工访问某些网站）、匿名化（隐藏客户端的真实IP）、负载均衡（将请求分发到多台服务器）。

6.4 电子邮件

为什么需要电子邮件协议？

发邮件看起来很简单——写好内容、点击发送、对方收到。但背后涉及三个不同阶段的问题：

  • 发送阶段：从你的电脑把邮件发到你的邮件服务器。用什么协议？SMTP。
  • 传输阶段：你的邮件服务器把邮件发送到对方的邮件服务器。也用SMTP。
  • 接收阶段：对方从他的邮件服务器把邮件取到自己的设备。用什么协议？POP3 或 IMAP。

注意一个关键区别：SMTP是推（Push）协议——主动把邮件推出去。POP3和IMAP是拉（Pull）协议——主动去服务器上取邮件。

电子邮件系统的三大组件

  • 用户代理（User Agent, UA）：用户撰写、阅读、管理邮件的工具。如Outlook、Thunderbird、Gmail Web界面。
  • 邮件服务器（Mail Server）：负责存储和转发邮件。发送方邮件服务器用SMTP发送，接收方邮件服务器用POP3/IMAP让用户取件。
  • 邮件协议：定义了邮件如何在组件之间传输。SMTP（发送）、POP3（接收-下载模式）、IMAP（接收-在线模式）。

SMTP：发送邮件

SMTP（Simple Mail Transfer Protocol，简单邮件传输协议）是电子邮件体系中最核心的协议。它只管"把邮件送到对方的邮件服务器"。

SMTP的特点：

  • 使用TCP 25端口（保证可靠传输）
  • 采用命令/应答交互方式，所有命令和响应都是ASCII文本（方便调试）
  • 只能传输7位ASCII文本（不能直接传二进制文件——这是MIME要解决的问题）

SMTP的通信过程分为三个阶段：

1

连接建立

发送方邮件服务器与接收方邮件服务器建立TCP连接（端口25）。接收方返回220（服务就绪），发送方发送HELO/EHLO命令表明身份。

2

邮件传送

发送方依次发送：MAIL FROM（发件人地址）、RCPT TO（收件人地址）、DATA（邮件内容）。每个命令都得到接收方的确认后，才执行下一步。DATA命令后，发送方发送完整的邮件内容（包括首部和主体），以单独一行的"点"（.）结束。

3

连接释放

发送方发送QUIT命令，接收方回复221（服务关闭），释放TCP连接。

POP3：接收邮件（下载模式）

POP3（Post Office Protocol version 3，邮局协议第3版）是最早的邮件接收协议之一。它模仿了"去邮局取信"的模式：

  • 使用TCP 110端口
  • 用户连接到邮件服务器，输入用户名和密码进行认证
  • 将服务器上的邮件下载到本地设备
  • 下载后，通常删除服务器上的邮件副本
  • 邮件在本地阅读，支持离线使用

POP3的缺点很明显：邮件只保存在你下载的那台设备上。如果你用电脑下载了邮件，手机上就看不到。这对于今天多设备并用的使用习惯来说，很不方便。

IMAP：接收邮件（在线模式）

IMAP（Internet Message Access Protocol，互联网消息访问协议）是更现代的邮件接收协议：

  • 使用TCP 143端口
  • 邮件保留在服务器上不被删除
  • 客户端可以在线浏览、搜索、管理服务器上的邮件
  • 支持文件夹管理：可以创建、删除、重命名服务器上的文件夹
  • 支持部分下载：可以先只下载邮件头（发件人、主题等），需要时再下载全文和附件
  • 多设备同步：电脑和手机上都看到同样的邮件状态（已读、已回复、已删除等）
一句话总结SMTP/POP3/IMAP的关系：SMTP是"送信员"——把信送到对方的邮箱所在的邮局。POP3是"取信人"——把信从邮局取走（邮局不留底）。IMAP是"在线邮箱管理员"——直接在邮局里看信、整理信，邮局始终保存着所有信件。

电子邮件格式

一封电子邮件由三部分构成：

  1. 信封：SMTP传输时使用的信息，包括发件人地址和收件人地址（在SMTP的MAIL FROM和RCPT TO命令中指定）
  2. 首部：包含From（发件人）、To（收件人）、Subject（主题）、Date（日期）、CC（抄送）、BCC（密送）等字段
  3. 主体：邮件的实际内容

MIME：突破SMTP的ASCII限制

SMTP只能传输7位ASCII文本——这意味着中文、图片、附件、二进制文件统统不能直接传输。MIME（Multipurpose Internet Mail Extensions，多用途互联网邮件扩展）解决了这个问题。

MIME不是替代SMTP，而是对SMTP的扩展。它在邮件首部中增加额外的字段，告诉接收方：

  • MIME-Version: 1.0 — 告诉接收方这是MIME邮件
  • Content-Type: text/plain; charset=UTF-8 — 说明内容的类型和编码
  • Content-Transfer-Encoding: base64 — 说明内容是如何编码的

MIME使用了Base64编码：把二进制数据（如图片）转换成由64个可打印ASCII字符组成的文本串。SMTP可以传输这些文本串，接收方再解码还原成原始数据。

MIME还支持多部分邮件（Multipart Messages）：一封邮件可以同时包含纯文本版本、HTML版本和图片附件。接收方根据邮件客户端的支持情况选择显示哪个版本。

6.5 DHCP：动态主机配置协议

为什么需要DHCP？

想象一下：你带着笔记本电脑到咖啡厅，连上Wi-Fi，然后就可以上网了。你不需要手动输入IP地址、子网掩码、默认网关、DNS服务器——所有这些配置都是自动完成的。

如果没有DHCP，每一台连接到网络的设备都需要网络管理员手动配置：

  • 分配一个不重复的IP地址
  • 配置正确的子网掩码
  • 配置默认网关的IP地址
  • 配置DNS服务器的IP地址

在一个只有几台设备的小型网络中，手动配置是可行的。但在一个大学校园网中（数千台设备），或者在一个咖啡厅的公共Wi-Fi中（每天有几百个陌生设备连接），手动配置就是灾难。DHCP（Dynamic Host Configuration Protocol）就是为解决这个问题而生的——让设备接入网络时自动获取所需的网络配置。

DORA：DHCP的四步工作流程

DHCP的工作过程包含四个步骤，缩写为DORA（Discover → Offer → Request → Acknowledge）：

D

Discover（发现）

客户端（新接入的设备）还不知道DHCP服务器在哪，也不知道自己的IP地址。它广播发送一个DHCP Discover报文（目的IP=255.255.255.255，这是广播地址），相当于在喊："这里有设备需要IP地址，有DHCP服务器吗？"

O

Offer（提供）

网络中的DHCP服务器收到Discover报文后，从自己的IP地址池中挑选一个可用的IP地址，广播发送DHCP Offer报文："我是DHCP服务器，我给你分配这个IP地址：192.168.1.100，子网掩码：255.255.255.0，默认网关：192.168.1.1，DNS：8.8.8.8"。

R

Request（请求）

客户端可能收到多个DHCP服务器的Offer（如果有多个DHCP服务器）。客户端从中选择一个（通常是第一个到达的），然后广播发送DHCP Request报文："我接受192.168.1.100这个地址，我选择的是这个DHCP服务器"。为什么要广播？为了让所有DHCP服务器都知道——选中的那个服务器准备分配，没被选中的可以收回Offer。

A

Acknowledge（确认）

被选中的DHCP服务器广播发送DHCP ACK报文："确认分配，192.168.1.100现在归你了！" 客户端收到后，开始使用这个IP地址和其他配置参数。

为什么前三步都要用广播？因为Discover阶段客户端还没有IP地址（也不知道DHCP服务器的地址），所以只能用广播（255.255.255.255）来"喊话"。Offer阶段服务器用广播因为客户端还没有IP地址。Request和Acknowledge用广播是为了通知所有DHCP服务器（确保没被选中的服务器把Offer回收回去）。

租期管理

DHCP分配的IP地址不是永久占用的——因为网络中的设备数量通常远大于可分配的IP地址数量（比如一个咖啡厅只有254个可用的私有IP地址，但一天可能有上千个设备连接）。DHCP使用租约（Lease）机制来管理IP地址的分配：

  • 每个分配的IP地址有一个租用期限（通常24小时~7天）
  • 当租期过半（50%）时，客户端会向原DHCP服务器发送单播的Request续租
  • 如果续租失败（原服务器没回应），在租期过87.5%时，客户端广播Request向任何DHCP服务器续租
  • 租约到期后，客户端必须释放该IP地址，重新执行DORA流程

这种机制既保证了IP地址的高效利用（不用的地址会被回收），又保证了设备的持续连接（在租期内可以续租，不会断网）。

6.6 其他应用层协议

6.6.1 Telnet

Telnet是互联网上最早的远程登录协议之一，允许用户通过网络登录到远程主机并在远程主机上执行命令。

Telnet的重要安全缺陷：Telnet的所有数据（包括用户名和密码）都以明文传输。在今天的网络环境中，这是不可接受的。因此几乎所有的远程登录需求都已经被SSH（Secure Shell，使用22端口，加密传输）取代。目前Telnet主要用于网络设备（如交换机、路由器）的本地调试。

Telnet的特点：

  • 使用TCP 23端口
  • 客户/服务器模式
  • 使用NVT（Network Virtual Terminal，网络虚拟终端）格式屏蔽不同终端设备的差异

6.6.2 SNMP

SNMP（Simple Network Management Protocol，简单网络管理协议）是网络管理的"眼睛和耳朵"——让网络管理员能够监控和管理网络中的各种设备（路由器、交换机、服务器、打印机等）。

SNMP的工作方式：

  • 使用UDP 161端口（管理器发送查询）和162端口（设备发送Trap通知）
  • 基于MIB（Management Information Base，管理信息库）——每个被管理的设备都有一个MIB，存储了该设备的各种状态信息（CPU使用率、接口流量、运行时间等）
  • 两种工作模式：轮询（Polling）——管理器定期查询设备的状态；陷阱（Trap）——设备在发生异常时主动通知管理器

  • FTP（21+20端口，TCP）：控制连接（21，传命令）+数据连接（20或随机端口，传数据）；主动模式 vs 被动模式（谁发起数据连接）；匿名FTP；TFTP（UDP，69端口）。
