UI 系统

当模组开始拥有自己的面板、条形 HUD 和交互窗口时，它就不再只是“世界内容”，而是在改写玩家的信息入口。

UI 的价值 很多模组机制并不适合只靠聊天文本或 Buff 图标表达。资源条、专属技能槽、合成面板、状态窗口，这些都需要 UI 系统来承载。

tModLoader UI 的基本架构

tModLoader 的 UI 通常围绕几个核心类展开：UIState 负责组织一个界面树，UserInterface 负责管理当前显示的状态，ModSystem 则常用来控制何时显示、何时更新、何时插入到主界面层。

UIState

你可以把 UIState 理解成“一个完整界面页面”或“一个面板根节点”。它下面可以挂文本、图片、按钮、滚动条、物品槽等各种 UI 元素。

一个最基础的 UIState

using Terraria.GameContent.UI.Elements;
using Terraria.UI;

namespace ExampleMod.Common.UI;

public class StarInfoUI : UIState
{
    public UIPanel Panel;
    public UIText Label;

    public override void OnInitialize()
    {
        Panel = new UIPanel();
        Panel.Left.Set(400f, 0f);
        Panel.Top.Set(200f, 0f);
        Panel.Width.Set(260f, 0f);
        Panel.Height.Set(120f, 0f);

        Label = new UIText("星辉能量：0");
        Label.Left.Set(16f, 0f);
        Label.Top.Set(16f, 0f);

        Panel.Append(Label);
        Append(Panel);
    }
}

这就是最基础的一块 UI：一个面板，里面放一段文字。看起来简单，但已经包含了 UI 系统最核心的几个概念：根状态、子元素、位置和尺寸。

布局系统：Left / Top / Width / Height

tModLoader UI 的布局不是网页式流布局，而是显式坐标 + 百分比混合布局。你经常会看到这样的写法：

Panel.Left.Set(400f, 0f);
Panel.Top.Set(200f, 0f);
Panel.Width.Set(260f, 0f);
Panel.Height.Set(120f, 0f);

每个 Set 都有两个值：像素偏移和百分比偏移。最常见的入门写法是像上面这样，用纯像素定位。等你做响应式或居中布局时，再更多利用百分比。

入门建议 刚开始做 UI，先用纯像素布局把界面跑起来。等逻辑稳定之后，再慢慢调整为更优雅的相对定位。不要一开始就把自己困在复杂布局调试里。

用 ModSystem 管理 UI 生命周期

单有 UIState 还不够，你还需要一个地方创建它、持有它，并在界面层里更新与绘制。这个职责通常放在 ModSystem 中。

using Microsoft.Xna.Framework;
using Terraria;
using Terraria.GameContent.UI.Elements;
using Terraria.UI;
using Terraria.ModLoader;
using System.Collections.Generic;

namespace ExampleMod.Common.Systems;

public class UISystem : ModSystem
{
    internal UserInterface StarInterface;
    internal Common.UI.StarInfoUI StarUI;

    public override void Load()
    {
        StarUI = new Common.UI.StarInfoUI();
        StarUI.Activate();
        StarInterface = new UserInterface();
        StarInterface.SetState(StarUI);
    }

    public override void UpdateUI(GameTime gameTime)
    {
        StarInterface?.Update(gameTime);
    }

    public override void ModifyInterfaceLayers(List<GameInterfaceLayer> layers)
    {
        int inventoryIndex = layers.FindIndex(layer => layer.Name.Equals("Vanilla: Inventory"));
        if (inventoryIndex != -1)
        {
            layers.Insert(inventoryIndex, new LegacyGameInterfaceLayer(
                "ExampleMod: Star UI",
                delegate
                {
                    StarInterface.Draw(Main.spriteBatch, new GameTime());
                    return true;
                },
                InterfaceScaleType.UI));
        }
    }
}

这里最关键的入口是 ModifyInterfaceLayers。它让你把自定义 UI 插进 Terraria 原版界面的某一层附近。插在不同层位，显示效果也会不同。

注意 UI 层级位置很重要。放错层，界面可能被背包遮住、压在地图下面，或者挡住原版重要 UI。通常要根据你的界面性质决定是更靠前还是更靠后。

动态更新文本

UI 如果只显示静态文字，那只是装饰。大多数情况下，它应该反映玩家当前状态，比如资源值、技能冷却、Boss 机制、区域信息。

public override void Update(GameTime gameTime)
{
    base.Update(gameTime);
    Player player = Main.LocalPlayer;
    int energy = player.GetModPlayer<StarPlayer>().starEnergy;
    Label.SetText($"星辉能量：{energy}");
}

这样一来，界面上的文本就会随着玩家状态实时变化。这个模式非常常用，尤其适合做专属资源条或状态面板。

常用 UI 元素

  • UIPanel：最常见的容器面板
  • UIText：文本显示
  • UIImage：图片
  • UIScrollbar：滚动条
  • UIList：纵向列表
  • UIElement：自定义通用元素基类

很多复杂 UI，本质上也是这些基础组件的组合。你不需要一开始就会所有元素，只要先掌握容器 + 文字 + 图片 + 自定义绘制，就能做出很大一部分功能面板。

按钮与交互事件

如果你希望 UI 可点击，比如切换模式、打开分页、确认操作，就需要处理事件。

using Terraria.GameContent.UI.Elements;
using Terraria.UI;

UITextPanel<string> button = new UITextPanel<string>("切换模式");
button.Left.Set(16f, 0f);
button.Top.Set(56f, 0f);
button.Width.Set(100f, 0f);
button.Height.Set(30f, 0f);
button.OnLeftClick += OnToggleClicked;
Panel.Append(button);

private void OnToggleClicked(UIMouseEvent evt, UIElement listeningElement)
{
    Main.NewText("已切换模式");
}

按钮其实也是 UI 元素的一种。你可以通过点击事件触发任意游戏逻辑，比如开关技能模式、请求网络同步、打开分页详情。

HUD：更轻量的信息展示

并不是所有信息都需要做成一个大面板。有些内容更适合作为 HUD，贴在血蓝条附近或屏幕角落，比如自定义资源条、特殊状态图标、Boss 机制提示。

using Microsoft.Xna.Framework;
using Microsoft.Xna.Framework.Graphics;
using Terraria;
using Terraria.GameContent;
using Terraria.UI;

public class StarEnergyBar : UIElement
{
    public override void Draw(SpriteBatch spriteBatch)
    {
        base.Draw(spriteBatch);

        Player player = Main.LocalPlayer;
        float current = player.GetModPlayer<StarPlayer>().starEnergy;
        float max = 100f;
        float fill = current / max;

        CalculatedStyle dims = GetDimensions();
        Rectangle back = new Rectangle((int)dims.X, (int)dims.Y, (int)dims.Width, (int)dims.Height);
        Rectangle bar = new Rectangle(back.X + 2, back.Y + 2, (int)((back.Width - 4) * fill), back.Height - 4);

        spriteBatch.Draw(TextureAssets.MagicPixel.Value, back, Color.Black * 0.75f);
        spriteBatch.Draw(TextureAssets.MagicPixel.Value, bar, new Color(90, 170, 255));
    }
}

这段示例展示了 HUD 的典型写法：自己继承 UIElement，在 Draw 里读取当前资源值，再用矩形或贴图画出填充条。为了把重点放在绘制逻辑本身，这里把最大值直接写成了 100f；真正项目里你通常会把最大资源值也放进 ModPlayer，再从那里统一读取。真正项目里你通常还会加边框、数字文本和颜色变化，但核心思路就是“在自定义元素里把状态直接可视化”。HUD 的设计原则和面板不同：面板强调内容完整，HUD 强调瞬时可读性。真正好的 HUD，玩家在战斗中扫一眼就知道关键信息，而不需要停下来读字。

点击展开：什么时候该用面板，什么时候该用 HUD

如果信息需要仔细阅读、包含多个按钮或列表，用面板；如果信息需要在战斗中持续可见、只传达 1-2 个关键指标，用 HUD。不要把所有内容都塞进屏幕常驻条，也不要把本该一眼可懂的内容藏进复杂窗口。

UI 设计最容易犯的错

  • 信息过多，战斗时根本看不过来。
  • 层级放错，被原版 UI 挡住或反挡原版功能。
  • 面板逻辑和游戏逻辑强耦合，难以维护。
  • 只追求好看，不考虑 Terraria 原生视觉风格和可读性。
最重要的原则 UI 不是展示你会不会做界面，而是帮助玩家更快理解模组机制。凡是不能提升理解效率的装饰，都应该谨慎添加。
